Claude Skills 가이드

Claude Code 스킬 시스템의 구조, 작성법, 베스트 프랙티스를 정리합니다

공식 문서 →
1

스킬이란?

스킬(Skill)은 Claude Code가 특정 명령(/skill-name)을 받았을 때 실행하는 재사용 가능한 프롬프트 템플릿입니다. 마크다운 파일에 프론트매터와 지시문을 작성하면, Claude Code가 이를 자동으로 인식하고 슬래시 명령으로 노출합니다.

⚡
즉시 실행
/commit 한 번으로 커밋 메시지 자동 생성 등 반복 작업을 즉시 실행
🔁
재사용 가능
한 번 작성하면 팀 전체가 동일한 동작을 공유. 프로젝트별 커스터마이징 지원
📦
점진적 컨텍스트
필요할 때만 파일을 읽어 컨텍스트 창 낭비 없이 관련 정보만 로드
🧩
조합 가능
스킬 안에서 다른 스킬을 호출하거나 파이프라인 형태로 연결 가능
2

파일 구조

스킬 파일은 .claude/skills/ 또는 프로젝트 루트의 .claude/commands/ 디렉토리에 위치합니다.

# 프로젝트 스킬 (팀 공유용) project-root/ ├── .claude/ │ ├── skills/ │ │ ├── commit.md # /commit 슬래시 명령 │ │ ├── review-pr.md # /review-pr 슬래시 명령 │ │ └── test-runner.md # /test-runner 슬래시 명령 │ └── CLAUDE.md # 프로젝트 전역 지시문 # 개인 전용 스킬 (글로벌) ~/.claude/ └── skills/ └── my-skill.md # 모든 프로젝트에서 사용 가능
3

SKILL.md 구성

스킬 파일은 YAML 프론트매터와 마크다운 본문(지시문)으로 구성됩니다.

--- name: commit description: 스테이징된 변경사항을 분석해 커밋 메시지를 자동 생성하고 커밋합니다 tools: - Read - Bash model: claude-haiku-4-5-20251001 --- # 커밋 지시문 1. `git diff --staged`로 변경 내용을 확인합니다 2. 변경 유형을 분류합니다: feat / fix / refactor / docs / chore 3. 한국어로 간결한 커밋 메시지를 작성합니다 4. `git commit -m "메시지"` 로 커밋을 실행합니다
4

Description 필드 — 핵심

description은 Claude가 언제 이 스킬을 호출할지 판단하는 유일한 근거입니다. 에이전트 모드에서 자동 라우팅되므로 정확하고 구체적으로 작성해야 합니다.

❌ 나쁜 예
description: 커밋 도구
description: 코드 리뷰
description: 테스트 실행
✅ 좋은 예
description: 스테이징된 변경사항을 분석해 한국어 커밋 메시지를 자동 생성하고 커밋합니다
description: PR의 diff를 읽고 보안·성능·가독성 관점에서 코드 리뷰 코멘트를 생성합니다

좋은 Description의 4요소

요소설명예시
대상 무엇을 입력으로 받는가 스테이징된 변경사항, PR의 diff, 테스트 파일
동작 무슨 작업을 하는가 분석, 생성, 실행, 검증
결과 무엇을 출력/수행하는가 커밋 메시지 작성, 리뷰 코멘트 생성
컨텍스트 어떤 규칙/제약이 있는가 (선택) 한국어로, Conventional Commits 형식으로
5

점진적 컨텍스트 로딩

Claude Code는 스킬 실행 시 모든 파일을 한 번에 로드하지 않습니다. 3단계 점진적 로딩으로 컨텍스트 창을 효율적으로 사용합니다.

단계로드 시점내용용도
STEP 1 항상 (대화 시작) 프론트매터 description 라우팅 결정 — 이 스킬을 쓸지 판단
STEP 2 스킬 호출 시 SKILL.md 전체 본문 지시문 실행 — 실제 작업 수행
STEP 3 지시문 내 명시 시 $INCLUDE 된 외부 파일 대용량 레퍼런스 (스타일 가이드, 스키마 등)
# $INCLUDE 활용 예시 코딩 컨벤션을 참고해 리뷰합니다: $INCLUDE .claude/docs/coding-conventions.md 위 가이드라인에 따라 아래 항목을 검토하세요:
6

프론트매터 필드 레퍼런스

필드필수타입설명
name 필수 string 슬래시 명령명. 파일명과 일치시키는 것을 권장
description 필수 string 에이전트 라우팅에 사용. 구체적이고 동작 중심으로 작성
tools 선택 string[] 허용 도구 목록. 미지정 시 모든 도구 허용
model 선택 string 스킬별 모델 오버라이드. 기본값: 현재 설정 모델
args 선택 string 스킬 실행 시 전달받는 인자 설명. $ARGUMENTS로 본문에서 참조
isolation 선택 string "worktree" 설정 시 Git worktree 격리 환경에서 실행
7

좋은 지시문 작성법

스킬 본문의 지시문은 Claude가 단계별로 따를 절차를 정의합니다. 5가지 원칙을 지키면 일관된 결과를 얻을 수 있습니다.

원칙 1 — 번호 순서로 단계 명시
Claude는 번호가 매겨진 목록을 순서대로 따릅니다. 순서가 중요한 작업은 반드시 번호 목록으로 작성하세요.
1. `git status`로 변경 파일 목록 확인 2. `git diff --staged`로 변경 내용 분석 3. 커밋 유형 결정 (feat/fix/refactor/docs) 4. 커밋 메시지 작성 및 실행
원칙 2 — 출력 형식 명시
결과물의 언어, 포맷, 길이를 명시적으로 지정하면 일관성이 높아집니다.
# 출력 형식 지정 예시 응답은 **한국어**로 작성합니다. 리뷰 코멘트는 다음 형식을 사용합니다: - 🔴 필수 수정: [내용] - 🟡 개선 권장: [내용] - 🟢 좋은 점: [내용]
원칙 3 — 조건 분기 명시
예외 상황을 미리 정의하면 Claude가 불확실한 상황에서도 올바르게 행동합니다.
스테이징된 변경사항이 없으면: → "커밋할 내용이 없습니다"를 출력하고 종료 테스트 파일만 변경된 경우: → "test:" 접두어를 사용합니다
원칙 4 — $ARGUMENTS 활용
사용자가 슬래시 명령과 함께 전달한 텍스트를 $ARGUMENTS로 참조합니다.
--- args: PR 번호 또는 브랜치명 (선택) --- # 사용: /review-pr 123 또는 /review-pr feature/auth 리뷰 대상: $ARGUMENTS $ARGUMENTS가 없으면 현재 브랜치를 기준으로 합니다.
원칙 5 — 확인 게이트 추가
되돌릴 수 없는 작업(배포, 삭제 등) 전에 사용자 확인 단계를 명시합니다.
배포 전 반드시 확인을 요청합니다: "⚠️ [환경]에 배포하시겠습니까? (y/n)" 사용자가 'y'를 입력했을 때만 계속합니다.
8

고급 패턴

패턴 A — 스킬 체이닝
한 스킬의 결과를 다음 스킬의 입력으로 연결합니다. Agent SDK의 파이프라인과 유사한 구조입니다.
--- name: ship description: 테스트 실행 → 커밋 → PR 생성을 순서대로 수행합니다 --- 1. /test-runner 를 실행하고 모든 테스트가 통과하면 계속합니다 2. /commit 을 실행해 변경사항을 커밋합니다 3. /create-pr 을 실행해 PR을 생성합니다
패턴 B — Worktree 격리 실행
위험한 작업은 별도 Git worktree에서 실행해 메인 브랜치를 보호합니다.
--- name: safe-refactor description: 격리된 환경에서 리팩토링을 수행하고 결과를 PR로 제출합니다 isolation: worktree --- 격리된 worktree에서 리팩토링을 수행합니다. 완료 후 변경사항을 커밋하고 PR 링크를 반환합니다.
패턴 C — 조건부 도구 허용
스킬마다 허용 도구를 제한해 의도치 않은 부작용을 방지합니다.
--- name: analyze-only description: 코드베이스를 분석하고 보고서를 생성합니다. 파일을 수정하지 않습니다 tools: # 읽기 전용 도구만 허용 - Read - Glob - Grep ---
9

테스트 & 이터레이션

스킬은 작성 후 반드시 검증이 필요합니다. 아래 5단계 프로세스로 안정적인 스킬을 만드세요.

1
초안 작성

최소한의 지시문으로 시작합니다. 처음부터 완벽하게 만들려 하지 마세요.

2
실제 환경에서 실행

테스트 레포지토리가 아닌 실제 프로젝트에서 /skill-name으로 실행합니다.

3
결과 분석

의도한 동작과 다른 부분을 기록합니다. 특히 누락된 단계, 잘못된 형식, 예외 미처리를 확인합니다.

4
지시문 구체화

모호한 표현을 명확하게 수정합니다. 반복되는 오류 패턴은 명시적 제약으로 추가합니다.

5
3회 이상 안정 확인

동일한 입력에 대해 일관된 결과가 나오면 팀에 공유합니다. Git에 커밋해 버전 관리합니다.

10

10가지 핵심 규칙

1
description은 동작 중심으로 — "무엇을 하는가"를 동사로 시작해 명확히 서술합니다
2
파일명 = 명령명 — commit.md는 /commit으로 호출됩니다. 일관성을 유지하세요
3
단일 책임 원칙 — 한 스킬은 한 가지 일만 합니다. 여러 작업은 체이닝으로 연결하세요
4
도구 최소 허용 — 필요한 도구만 tools에 나열해 부작용을 방지합니다
5
출력 언어 명시 — 한국어/영어 혼용을 막으려면 응답 언어를 지시문에 명시하세요
6
확인 게이트 필수 — 배포, 삭제 등 되돌릴 수 없는 작업 전에 반드시 확인을 요청합니다
7
$INCLUDE 적극 활용 — 대용량 레퍼런스는 별도 파일로 분리해 컨텍스트 창을 절약합니다
8
예외 상황 명시 — "~이 없으면", "~인 경우" 분기를 지시문에 명확히 정의합니다
9
Git으로 버전 관리 — 스킬 파일도 코드입니다. 변경 이력을 커밋으로 추적하세요
10
실사용 후 이터레이션 — 실제 환경에서 3회 이상 안정 확인 후 팀에 공유합니다
